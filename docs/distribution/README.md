# Package Manager Publishing

This document captures what Cosha needs before publishing through Linux, macOS, and Windows package managers.

## Release Artifacts

Publish immutable artifacts from GitHub Releases before submitting package manager manifests.

- Use signed Git tags that match the release version, such as `v26.06.0`.
- Build from a clean release tag, not from a branch.
- Produce archives for each supported OS and CPU pair:
  - `darwin/arm64`
  - `darwin/amd64`
  - `linux/arm64`
  - `linux/amd64`
  - `windows/arm64`
  - `windows/amd64`
- Use predictable names such as `cosha_26.06.0_darwin_arm64.tar.gz` and `cosha_26.06.0_windows_amd64.zip`.
- Include only the executable, license, README, and shell completions when they exist.
- Publish SHA-256 checksums for every artifact.
- Sign checksums or artifacts before package manager submission.
- Keep release notes aligned with `CHANGELOG.md`.

## Build Constraints

Cosha currently depends on Tree-sitter through CGO. Treat release builds as native platform builds unless cross-compilation is proven and automated.

- Build macOS artifacts on macOS.
- Build Windows artifacts on Windows.
- Build Linux artifacts on Linux.
- Verify `cosha --help`, `cosha detect`, and a small index/search smoke test on every release artifact.
- Avoid package-manager formulas that rebuild from source unless the build environment has the required Go version, C compiler, and CGO support.
- Prefer binary packages first; add source-building packages later if needed.

## Install Layout

Package managers should install only the `cosha` executable and optional static assets such as completions.

- Install the binary into the package manager's normal executable path.
- Do not install generated indexes, `.cosha/`, or sample repository data.
- Keep runtime state local to the user or repository.
- Ensure uninstall removes the binary but does not remove user-created indexes.
- Confirm `cosha` is available on `PATH` after install.

## Metadata

Keep package metadata consistent across managers.

- Package ID: `cosha`
- Executable: `cosha`
- License: `Apache-2.0`
- Homepage: project repository URL
- Source: project repository URL
- Description: local-first codebase intelligence for coding agents
- Version: match the GitHub release tag without package-manager-specific suffixes unless required
- Dependencies: avoid runtime dependencies where possible; the released binary should be self-contained for the target platform

## macOS And Linux: Homebrew

Homebrew can cover macOS and Linux through a tap.

- Create a tap repository such as `homebrew-tap`.
- Add a `cosha.rb` formula that downloads release artifacts and verifies SHA-256 checksums.
- Prefer prebuilt bottles once the release process is stable.
- Test locally with `brew install --build-from-source` or an artifact-backed formula before submitting.
- Confirm Apple Silicon installs under `/opt/homebrew` and Intel macOS installs under `/usr/local`.

References:

- Homebrew Formula Cookbook: https://docs.brew.sh/Formula-Cookbook
- Homebrew taps: https://docs.brew.sh/Taps

## Linux Packages

Start with downloadable archives plus one broad package channel. Add distro-native packages after release automation is stable.

Recommended order:

1. GitHub Release archives for `linux/amd64` and `linux/arm64`.
2. Homebrew on Linux for users who already use Homebrew.
3. `.deb` and `.rpm` packages generated by release automation.
4. Snap only if confinement and store review are worth the maintenance cost.

For `.deb` and `.rpm` packages:

- Install `cosha` into `/usr/bin` or the package-manager-approved equivalent.
- Include license and README in the package documentation path.
- Do not create or own user runtime directories.
- Run package linting before publishing.

References:

- Debian binary package policy: https://www.debian.org/doc/debian-policy/ch-binary.html
- Snapcraft documentation: https://snapcraft.io/docs

## Windows Packages

Support Windows with a `.zip` release artifact first, then add package manager manifests.

Recommended order:

1. GitHub Release `.zip` artifacts for `windows/amd64` and `windows/arm64`.
2. Scoop manifest.
3. winget manifest.
4. Chocolatey package if the project wants Chocolatey's moderation and package conventions.

For Windows package managers:

- Ensure the archive contains `cosha.exe`.
- Prefer user-level install flows unless a manager expects machine-level install.
- Verify install, upgrade, and uninstall behavior in a clean Windows environment.
- If signing is added, use Authenticode signing for Windows binaries.

References:

- winget manifests: https://learn.microsoft.com/en-us/windows/package-manager/package/manifest
- winget repository submission: https://learn.microsoft.com/en-us/windows/package-manager/package/repository
- Scoop manifests: https://github.com/ScoopInstaller/Scoop/wiki/App-Manifests
- Scoop autoupdate: https://github.com/ScoopInstaller/Scoop/wiki/App-Manifest-Autoupdate
- Chocolatey package creation: https://docs.chocolatey.org/en-us/create/create-packages/

## Automation

Use release automation once manual release artifacts are proven.

- Generate archives, checksums, and package manifests from one release workflow.
- Keep the Go version pinned to the repo's `go.mod` requirement.
- Run `make check` before release builds.
- Build each CGO target on its native OS runner.
- Publish artifacts only after tests pass.
- Consider GoReleaser for archives, checksums, Homebrew taps, Scoop manifests, and Linux packages.

Reference:

- GoReleaser signing and artifact documentation: https://goreleaser.com/customization/sign/

## Release Checklist

Before publishing to package managers:

- [ ] Release tag is signed and pushed.
- [ ] `CHANGELOG.md` has the released version and date.
- [ ] Artifacts exist for every advertised OS/arch pair.
- [ ] Checksums are published and signed.
- [ ] Artifacts pass smoke tests on macOS, Linux, and Windows.
- [ ] Package metadata uses the same version and license everywhere.
- [ ] Install, upgrade, and uninstall are tested for each package manager.
- [ ] Package submissions link to immutable GitHub Release artifacts.
- [ ] Maintainer credentials, signing keys, and automation tokens are documented outside the repository.
